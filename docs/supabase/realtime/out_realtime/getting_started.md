### Get API details<a href="https://supabase.com/docs/guides/realtime/getting_started#get-api-details" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

Now that you've created some database tables, you are ready to insert data using the auto-generated API.

To do this, you need to get the Project URL and key. Get the URL from [the API settings section](https://supabase.com/dashboard/project/_/settings/api) of a project and the key from the [the API Keys section of a project's Settings page](https://supabase.com/dashboard/project/_/settings/api-keys/).

<img src="out_realtime/getting_started/index_media/0f7fc93538e91ade401009300159ebc5054e285d.svg" class="w-6 h-6" />

##### Changes to API keys

Supabase is changing the way keys work to improve project security and developer experience. You can [read the full announcement](https://github.com/orgs/supabase/discussions/29260), but in the transition period, you can use both the current `anon` and `service_role` keys and the new publishable key with the form `sb_publishable_xxx` which will replace the older keys.

To get the key values, open [the API Keys section of a project's Settings page](https://supabase.com/dashboard/project/_/settings/api-keys/) and do the following:

- **For legacy keys**, copy the `anon` key for client-side operations and the `service_role` key for server-side operations from the **Legacy API Keys** tab.
- **For new keys**, open the **API Keys** tab, if you don't have a publishable key already, click **Create new API Keys**, and copy the value from the **Publishable key** section.

### 4. Set up authorization<a href="https://supabase.com/docs/guides/realtime/getting_started#4-set-up-authorization" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

Since we're using a private channel, you need to create a basic RLS policy on the `realtime.messages` table to allow authenticated users to connect. Row Level Security (RLS) policies control who can access your Realtime channels based on user authentication and custom rules:

``` flex
1234567-- Allow authenticated users to receive broadcastsCREATE POLICY "authenticated_users_can_receive" ON realtime.messages  FOR SELECT TO authenticated USING (true);-- Allow authenticated users to send broadcastsCREATE POLICY "authenticated_users_can_send" ON realtime.messages  FOR INSERT TO authenticated WITH CHECK (true);
```

#### 5.3 using database triggers<a href="https://supabase.com/docs/guides/realtime/getting_started#53-using-database-triggers" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

Automatically broadcast database changes using triggers. Choose the approach that best fits your needs:

**Using `realtime.broadcast_changes` (Best for mirroring database changes)**

``` flex
12345678910111213141516171819202122-- Create a trigger function for broadcasting database changesCREATE OR REPLACE FUNCTION broadcast_message_changes()RETURNS TRIGGER AS $$BEGIN  -- Broadcast to room-specific channel  PERFORM realtime.broadcast_changes(    'room:' || NEW.room_id::text || ':messages',    TG_OP,    TG_OP,    TG_TABLE_NAME,    TG_TABLE_SCHEMA,    NEW,    OLD  );  RETURN NULL;END;$$ LANGUAGE plpgsql SECURITY DEFINER;-- Apply trigger to your messages tableCREATE TRIGGER messages_broadcast_trigger  AFTER INSERT OR UPDATE OR DELETE ON messages  FOR EACH ROW EXECUTE FUNCTION broadcast_message_changes();
```

<img src="out_realtime/getting_started/index_media/d9c4b7add2b04096741724568af26673b4f446a9.svg" class="lucide lucide-copy text-lighter" />

**Using `realtime.send` (Best for custom notifications and filtered data)**

``` flex
123456789101112131415161718192021222324252627-- Create a trigger function for custom notificationsCREATE OR REPLACE FUNCTION notify_message_activity()RETURNS TRIGGER AS $$BEGIN  -- Send custom notification when new message is created  IF TG_OP = 'INSERT' THEN    PERFORM realtime.send(      'room:' || NEW.room_id::text || ':notifications',      'message_created',      jsonb_build_object(        'message_id', NEW.id,        'user_id', NEW.user_id,        'room_id', NEW.room_id,        'created_at', NEW.created_at      ),      true  -- private channel    );  END IF;  RETURN NULL;END;$$ LANGUAGE plpgsql SECURITY DEFINER;-- Apply trigger to your messages tableCREATE TRIGGER messages_notification_trigger  AFTER INSERT ON messages  FOR EACH ROW EXECUTE FUNCTION notify_message_activity();
```

<img src="out_realtime/getting_started/index_media/d9c4b7add2b04096741724568af26673b4f446a9.svg" class="lucide lucide-copy text-lighter" />

- **`realtime.broadcast_changes`** sends the full database change with metadata
- **`realtime.send`** allows you to send custom payloads and control exactly what data is broadcast

## Essential best practices<a href="https://supabase.com/docs/guides/realtime/getting_started#essential-best-practices" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

### Use private channels<a href="https://supabase.com/docs/guides/realtime/getting_started#use-private-channels" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

Always use private channels for production applications to ensure proper security and authorization:

``` flex
123const channel = supabase.channel('room:123:messages', {  config: { private: true },})
```

<img src="out_realtime/getting_started/index_media/d9c4b7add2b04096741724568af26673b4f446a9.svg" class="lucide lucide-copy text-lighter" />

### Follow naming conventions<a href="https://supabase.com/docs/guides/realtime/getting_started#follow-naming-conventions" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

**Channel Topics:** Use the pattern `scope:id:entity`

- `room:123:messages` - Messages in room 123
- `game:456:moves` - Game moves for game 456
- `user:789:notifications` - Notifications for user 789

### Clean up subscriptions<a href="https://supabase.com/docs/guides/realtime/getting_started#clean-up-subscriptions" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

Always unsubscribe when you are done with a channel to ensure you free up resources:

Flutter

Swift

Python

``` flex
12345678910// React exampleimport { useEffect } from 'react'useEffect(() => {  const channel = supabase.channel('room:123:messages')  return () => {    supabase.removeChannel(channel)  }}, [])
```

## Choose the right feature<a href="https://supabase.com/docs/guides/realtime/getting_started#choose-the-right-feature" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

### When to use Broadcast<a href="https://supabase.com/docs/guides/realtime/getting_started#when-to-use-broadcast" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

- Real-time messaging and notifications
- Custom events and game state
- Database change notifications (with triggers)
- High-frequency updates (e.g. Cursor tracking)
- Most use cases

### When to use Presence<a href="https://supabase.com/docs/guides/realtime/getting_started#when-to-use-presence" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

- User online/offline status
- Active user counters
- Use minimally due to computational overhead

### When to use Postgres Changes<a href="https://supabase.com/docs/guides/realtime/getting_started#when-to-use-postgres-changes" class="ml-2 opacity-0 group-hover:opacity-100 transition" aria-hidden="true">#</a>

- Quick testing and development
- Low amount of connected users
